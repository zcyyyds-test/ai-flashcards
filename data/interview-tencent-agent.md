## Q1：Skill 太多塞爆 system prompt 怎么办？

### 核心答案
**不能把所有 skill 常驻 system prompt**，会带来三个问题：
1. 上下文窗口被占满
2. 候选技能噪声太大
3. 模型在选 skill 时更容易混淆

**更合理的方式**：把 skill 做成**外部注册表**，system prompt 里只保留最小规则和调用协议，真正的 skill 描述按需动态注入。

### 工程做法
**一层 skill routing**：
- 规则、分类模型或向量检索先筛出 top-k skill
- 再把这几个 skill 的 description、参数 schema 和 few-shot 示例拼进 prompt
- 如果 skill 本身很长，还可以拆成**摘要版和完整版**——先注入摘要，模型确定要调用后再加载详细说明

### 延伸
- **向量检索 skill** 和 RAG 本质一样：skill description 当文档，query 当问题
- **Skill 摘要** 可以在注册时用 LLM 离线生成一次，不用每次 in-context 压缩
- **缓存**：经常共现的 skill 组合可以预先缓存好 prompt 片段

## Q2：两个 skill description 相似导致选错怎么办？

### 核心答案
description 相似时单靠自然语言匹配很容易误召回。解决思路是**给 skill 增加更强的判别信息**，而不是继续堆模糊描述。

### 四步落地
1. **结构化定义**：从纯文本 description 升级成结构化 schema，包括
   - **适用场景 / 禁用场景**
   - 输入参数、返回格式
   - 前置条件、失败条件
   - 调用示例
2. **两阶段选择**：先召回 top-k（粗排），再让模型根据参数约束和样例做 **rerank**（精排）
3. **负样本示例**：明确告诉模型"**什么问题不应该调用这个 skill**"
4. **执行前校验**：schema 校验 + rule check，即使模型选错也不能继续误调

### 极端情况：语义长期接近
如果两个 skill 语义长期接近，通常要在**系统设计层面合并成一个 skill**，再由内部参数路由，否则前面再怎么写 prompt，后面也会持续混。

### 延伸
- **"禁用场景"** 是 prompt 工程里被低估的技巧——负面约束比正面描述更能降低误召回
- **Rerank** 这一步通常可以用更便宜的小模型，主模型只负责最终执行
- 这道题本质是在问 **"Agent 层的 typing/schema 设计"**，和传统软件工程的接口设计思路一致

## Q3：Agent 设计里最重要的部分是什么？

### 核心答案
**上下文工程（Context Engineering）**。

Agent 最终不是单轮问答，而是一个**持续决策系统**。模型能看到什么信息、这些信息的顺序和优先级是什么、哪些内容是规则、哪些是证据、哪些只是历史状态——这些都会直接影响 Agent 的稳定性。

### 核心论点
> Agent 设计的核心不是"给模型更多信息"，而是"**给模型正确的信息**"。

上下文工程做得好，模型会像一个**受控执行器**——知道当前目标、约束条件、可调用工具、历史状态和输出格式。

上下文工程做得差，模型即使能力强，也会出现：
- 目标漂移
- 工具误选
- 记忆污染
- 长对话失真
- 推理跑偏

### 延伸
- 这题是腾讯这一面的**灵魂题**——后面的 Q4、Q5、Q11、Q14 都在展开这个观点
- 和 Anthropic 最近提的 "Context Engineering" 方法论一致（2024 年开始 Prompt Engineering 被逐步替换为 Context Engineering）
- **反面例子**：把所有历史对话、所有检索结果、所有工具说明一股脑灌进去 → 模型不堪重负

## Q4：上下文工程有哪些要注意的点？

### 五个要点

**1. 角色隔离**
- system / developer / user / tool output / memory 的优先级和边界要清楚
- 外部文本和系统规则**不能混在一起**

**2. 信息裁剪**
- 只保留当前任务相关内容
- 避免把所有历史对话、所有检索结果和所有工具说明一股脑灌进去

**3. 格式控制**
- 结构化上下文通常比纯自然语言更稳
- 工具结果尽量用 JSON
- 任务状态单独字段化
- 长历史做摘要而不是堆原文

**4. 来源可信度分层**
- 系统规则 > 用户输入 > 工具结果 > 模型猜测
- 知识证据 > 无依据生成

**5. 长任务要做上下文压缩和过期淘汰**
- 越到后面越容易发生状态漂移和旧信息污染新决策

### 延伸
- **角色隔离**对应 OpenAI 的 "instruction hierarchy"（2024）和 Anthropic 的 "role isolation"
- **可信度分层**是 Prompt Injection 防御的底层原理（Q12）
- **JSON vs 自然语言**：JSON 对模型更稳，但 token 开销略大；腾讯这种大规模线上 Agent 选 JSON 是对的

## Q5：长期记忆的写回 / 衰减 / 冲突消解怎么设计？

### 核心答案
**长期记忆不能等价于"把所有历史都存起来"**。真正可用的长期记忆必须回答三个问题：
- **什么值得写回**（写回策略）
- **什么时候应该忘**（衰减策略）
- **冲突信息怎么处理**（消解策略）

### 写回策略
- 只保存**高价值稳定信息**：用户偏好、身份属性、长期任务目标、反复出现的业务事实
- **不是**每句对话都入库
- 写回时做**结构化抽取**：把"用户更喜欢用 Python"写成 profile 字段，而不是原文整段存储

### 衰减策略
记忆需要时效性和权重：
- **短期热点**信息可以高权重，但随时间衰减
- **长期稳定偏好**可以低频刷新但不轻易删除

### 冲突消解
- 结合**时间戳、来源可信度、置信度**
- **新的高可信事实覆盖旧事实**
- **低可信内容**只作为候选，不直接替换

### 进阶：事件流 + 物化视图
更严格的做法——把长期记忆设计成 **event sourcing + materialized view**：
- 所有记忆先写入 event store
- 再异步聚合成 profile
- 这样既方便审计，也方便回滚

### 延伸
- 这一题本质是把 **数据库 CDC / event sourcing** 的思想搬到 Agent Memory 上
- **物化视图**的好处：profile 被污染可以从 event 重新 rebuild，不用人工修数据
- 对比传统做法（直接 UPDATE memory 表）：event sourcing 的最大好处是"记忆可解释、可回滚"

## Q6：并行 Tool Calling 怎么既提升吞吐又保证一致性和可回放？

### 核心答案
并行 Tool Calling 的核心**不是"同时调多个接口"**，而是解决：
- **依赖关系**
- **状态一致性**
- **结果合并**
- **失败回滚**

### 四个关键点

**1. 并行的前提是无依赖**
- 只有互不依赖的工具才能并行（天气查询 和 汇率查询 可以一起发）
- 如果后一个工具依赖前一个工具的结果，就必须串行

**2. 状态一致性**
- 一般不让多个工具直接修改共享状态
- 采用 **"工具执行结果先写到临时观察区，再由调度器统一合并"**
- 这样可以避免并发写导致状态污染

**3. 可回放**
- 每次调用都带 **trace_id、step_id、输入参数、输出结果和耗时**
- 最后组成完整的**执行 DAG**
- 失败处理上可以按工具粒度重试，不能因为一个工具超时就把全部结果丢掉

**4. 副作用操作必须幂等**
- 发消息、创建订单、提工单这些副作用操作
- **并行执行前通常还要做幂等键**
- 不然重复重试会造成真实业务重复写入

### 延伸
- **DAG 记录**这件事可以复用 OpenTelemetry 或自研 trace，不必重造轮子
- **幂等键**设计是分布式系统里的老话题，但在 Agent 场景经常被忽略——LLM 不知道"重复 rollout 的代价"
- 对比普通微服务调用，Agent 多了一层"模型决定要不要并行"的不确定性，所以**执行器必须兜底判断**

## Q7：了解主流的 Agent 设计吗？

### 六类主流模式

**1. ReAct**（经典）
- 边推理边行动：Thought → Action → Observation → Thought → ...
- 模型通过思考、调用工具、观察结果、继续思考来完成任务
- **优点**：简单通用；**缺点**：容易原地打转

**2. Plan-and-Execute**
- 先做规划（生成计划），再按步骤执行
- 比纯 ReAct 更稳，适合**长任务**
- **优点**：可解释性强；**缺点**：计划一旦错难以纠正

**3. Router 模式**
- 先做任务分类，再路由给不同的子 Agent 或工具链
- **优点**：适合任务差异大的场景；**缺点**：路由分类本身是瓶颈

**4. Workflow + Agent**
- 固定流程里嵌局部自主决策
- **在线业务里用得很多**——业务 SOP 不变，关键决策点交给 Agent
- **优点**：可控性最高；**缺点**：灵活度低

**5. Supervisor 模式**（多 Agent）
- 主 Agent 拆解任务并分派给多个子 Agent
- 适合**复杂任务分解**

**6. Blackboard 模式**（多 Agent）
- 多个 Agent 通过共享状态协作
- 纯点对点自由协作也有，但线上比较少，因为不容易治理

### 延伸
- **Workflow + Agent** 是目前企业落地最多的形态——LLM 负责"软决策"，固定流程负责"硬约束"
- **Supervisor vs Blackboard**：前者中心化调度（LangGraph、AutoGen），后者去中心化（实验室场景多）
- **ReAct 的死循环**是常见 bug → 见 Q15

## Q8：MCP、A2A、Function Calling 的本质区别？

### 核心差异：抽象层级不同

| 协议 | 抽象层级 | 本质 |
|---|---|---|
| **Function Calling** | 调用一个函数 | 单机式工具调用协议 |
| **MCP** | 暴露一组标准能力 | 标准化的上下文和工具暴露协议 |
| **A2A** | 多个智能体协作 | Agent 与 Agent 之间的协作协议 |

### Function Calling
- 模型输出结构化调用意图，宿主进程负责执行函数并把结果返回
- **重点**：模型怎么调用本地或受控工具
- **场景**：单应用内的工具调用

### MCP（Model Context Protocol）
- 把**资源、工具、提示模板**等能力统一封装出来
- 解决的是"**外部能力如何以标准方式被模型客户端发现和调用**"
- **场景**：多个模型客户端或 IDE 共享一套工具和资源能力

### A2A（Agent-to-Agent）
- 更偏 Agent 与 Agent 之间的协作协议
- **重点不是"一个模型调一个函数"**，而是多个智能体之间如何交换任务、状态和结果
- **场景**：多 Agent 分工协作、任务派发、结果汇总

### 工程选型
- 单应用内工具调用 → **Function Calling** 足够
- 多客户端/IDE 共享工具 → **MCP** 合适
- 多 Agent 分工协作 → **A2A** 合适

### 延伸
- MCP 是 Anthropic 2024 底提的，目前生态增长最快
- A2A 是 Google 2025 年推的，还在发展期
- 本质上这三个是**不同抽象层的协议**，不是互斥关系——一个大型 Agent 系统可能同时用三种

## Q9：RAG 接到 Agent 里，一般怎么设计？

### 核心答案
RAG 进入 Agent 后，**不应该只是固定的"先查后答"**，而应该作为**一个决策节点**：
- 模型需要判断这个问题**是否真的需要外部访问**
- 如果需要，再决定 query rewrite、query decomposition、召回、rerank 和生成的顺序

### 常见做法
- 把 RAG **包成一个工具**，Agent 自己决定何时调用
- 检索结果不原封不动扔给模型，而是做一层**切片、注入和来源标注**
- 生成时通常还会要求模型**只根据检索到的证据回答**
- **没证据就明确说不确定**——这样才能真正降低幻觉

### 关键设计点
1. **RAG 是工具不是管道**：让 Agent 决定要不要查
2. **query rewrite**：用户问"我的订单呢"→ 重写成"用户 X 最近 7 天订单状态"
3. **decomposition**：多跳问题拆成多个子查询
4. **证据标注**：召回结果带 doc_id、段落 id、时间戳
5. **幻觉防御**：prompt 强制"只根据证据答，没证据说不确定"

### 延伸
- **不知道就说不知道**是降幻觉的核心思路，比微调更有效
- **Agentic RAG**（自主决定检索）是 2024-2025 的主流升级方向
- 对比传统 RAG：传统 RAG 是"每问必查"，Agentic RAG 是"看情况再查"

## Q10：Tool Calling 和普通函数调用的区别？

### 核心差异
- **普通函数调用**：程序写死的——谁调用哪个函数、传什么参数都由开发者控制
- **Tool Calling**：**模型根据任务上下文自主输出工具名和参数**——所以必须加一层校验和调度

### 工程四层架构
```
┌─────────────────────┐
│  1. 工具注册层       │  定义工具 schema、白名单
├─────────────────────┤
│  2. 参数 schema 层   │  校验模型输出的参数合法性
├─────────────────────┤
│  3. 执行器层         │  真正调用函数、处理超时和重试
├─────────────────────┤
│  4. 审计日志层       │  记录每次调用，用于回放和分析
└─────────────────────┘
```

### 关键原则
- 模型**只能输出候选调用**
- 服务端**必须做**：白名单校验、参数校验、超时控制、幂等控制、错误包装
- **真正上线时，不能把工具执行权直接交给模型**

### 延伸
- 这一题和 Q6（并行 Tool Calling）、Q12（Prompt Injection）是连着的——都是"**不能信任模型输出**"
- **四层架构**可以直接当做 Agent 平台的设计蓝本
- OpenAI Functions / Anthropic Tool Use / Gemini Function Calling 三家的 API 本质都是这个模型

## Q11：Agent 的 Memory 一般怎么做？

### 三层记忆

**1. 短期记忆**
- 保留最近几轮对话和当前任务状态
- 保证会话连贯
- 通常是 context window 内的 messages

**2. 长期记忆**
- 保存稳定事实：用户偏好、身份属性、长期任务目标
- 结构化存储（profile 字段 / 知识图谱）

**3. 摘要记忆**
- 用于长对话中压缩历史
- 把对话提炼成关键事件，减少 token 占用

### 落地原则
真正落地时不会把所有对话原文都塞进去，而是：
- **保留最近 N 轮原文**
- **更久远的内容压成 summary**
- **抽取关键字段单独存储**
- **生成时按需召回**——而不是无脑全量拼接

### 延伸
- 这一题和 Q5（记忆的写回/衰减/冲突）是配套的
- **摘要记忆**的主要风险是"摘要丢细节"→ 需要在 summary 里保留 event pointers 能回原文
- **长期记忆的存储介质**：向量库（Pinecone/Weaviate）、结构化库（Postgres JSONB）、图数据库（Neo4j）各有适用场景

## Q12：Prompt Injection 在 Agent 场景怎么防？

### 为什么 Agent 场景更危险
Prompt Injection 是把恶意指令混进用户输入或外部文档中，诱导模型忽略既定规则。**Agent 场景里危险更高**，因为它不仅会生成错误文本，还可能：
- 误调工具
- 越权执行
- 触发真实业务操作

### 五层防御

**1. 角色和优先级隔离**
- 外部知识**不能覆盖系统规则**
- 系统提示词和用户/工具输入严格分离

**2. 把检索文档视为证据，不视为命令**
- 即使文档里写"请忽略上面的规则"，也只是待查内容
- 绝不作为执行指令

**3. 所有工具调用都要白名单和参数校验**
- 高风险操作必须**显式确认**（sudo 机制）

**4. 输出和调用链路必须可追踪**
- 出问题能回放

**5. 输入清洗和风险分类**
- 对明显越权或诱导内容做拦截

### 核心原则
> **核心不是靠一句 prompt 防住，而是靠系统设计让模型即使被诱导，也没有越权能力。**

### 延伸
- 这是 Agent 安全最重要的一句话——**不要指望 prompt 护栏，要指望系统护栏**
- 对应 Anthropic Constitutional AI 和 OpenAI Instruction Hierarchy 的思路
- **sudo 模式**：高风险操作（转账、删除、发送）必须人工确认 → 即使模型被劫持也无法造成真实损失

## Q13：怎么评估一个 Agent 系统的效果？

### 四层评估

**第一层：回答质量**
- 准确性、完整性、是否基于证据

**第二层：工具质量**
- 选工具是否正确
- 参数是否正确
- 调用是否成功

**第三层：任务质量**
- 一次会话是否完成目标
- 是否需要额外追问
- 是否需要人工接管

**第四层：系统质量**
- 耗时、成本、token、错误率、稳定性

### RAG 专项
如果有 RAG，还要单独评估：
- 召回命中率
- rerank 效果
- 答案引用质量

### 核心观点
> **Agent 不是单个模型的分数，而是整条链路的分数。**

### 延伸
- 这一题考的是**系统化思维**——不是"跑个 benchmark 看个 accuracy" 就完事
- **工具选择准确率** 是最容易被忽略的指标——很多 Agent"答得挺好但工具链乱七八糟"
- **人工接管率**是真实业务里的金标准指标：低接管率 = 真上线

## Q14：线上 Agent 经常选错工具，怎么排查？

### 排查顺序

**1. 看 tool description 和 schema 设计**
- 边界是否模糊
- 参数是否能区分场景

**2. 看 skill routing 层**
- 是否一次性喂了过多候选工具
- 造成噪声竞争

**3. 看 prompt 和 few-shot 示例**
- 是否给了错误引导
- 最近是否改过 system prompt

**4. 回放 trace**
- 检查模型选工具前到底看到了哪些上下文
- 历史记忆有没有污染当前判断
- 用户输入是否歧义严重

### 核心方法论
> **排查通常不是看最后一步，而是要看"候选集是怎么来的、模型为什么这么选、执行前有没有二次校验"。**

### 延伸
- **三段排查法**：候选集 → 决策过程 → 执行校验
- 对应软件工程的"**input → process → output**" debug 范式
- 和 Q13 的"可追踪"配套——没 trace 就没办法排查

## Q15：长任务场景怎么避免 Agent 无限推理或死循环？

### 四类终止条件

**1. 状态机和终止条件**
实际系统里必须限制：
- **最大推理步数**
- **最大工具调用次数**
- **最大 token 开销**
- **最大执行时长**

**2. 明确的成功/失败定义**
- 什么叫**完成**
- 什么叫**失败**
- 什么情况需要让用户补信息

**3. 重复检测**
- 如果连续几步都调用同一个工具、得到相似结果或者没有带来新证据
- 应该**强制终止**或**返回阶段性结论**

**4. 无进展检测**
- 连续 N 步 plan 不变
- 连续 N 步 observation 不变
- 连续 N 步 reward 不变（如果有）

### 核心原则
> **这样才能避免 Agent 陷入无意义循环。**

### 延伸
- 这一题是 **Q7 里 ReAct 死循环的对应解法**
- **最大步数**对应传统算法里的 "iteration budget"
- **重复检测**可以用 embedding similarity 监测
- 真实线上：**硬性 token/time budget** 是最后的兜底——即使算法层漏了，预算也不会爆
